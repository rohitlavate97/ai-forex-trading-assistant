# Feature Guide - Economic Calendar Agent

This guide documents the database schema, query structures, FastAPI routers, Pydantic AI agent dependencies, prompts, and evaluation parameters for the Economic Calendar Specialist Agent.

---

## 1. Business Requirement

*   **Macroeconomic Intelligence**: Track key events (GDP, CPI, Central Bank rate decisions) that generate volatility spikes.
*   **Decoupled Database Entities**: Store event parameters in a structured SQL table (`economic_events`) rather than context prompts, enabling clean searches and updates.
*   **Information-Only Guardrails**: The agent analyzes and explains data but does not suggest active live positions. Every output ends with a warning disclaimer.

---

## 2. Financial & Domain Context

Economic releases affect currency values via market sentiment:
*   **Interest Rate Decisions**: Higher interest rates attract global capital, boosting that nation's currency.
*   **Consumer Price Index (CPI)**: Measures inflation. Higher inflation increases likelihood of interest rate hikes.
*   **Gross Domestic Product (GDP)**: Measures economic strength. Higher GDP is bullish for the domestic currency.
*   **Forecast vs. Actual Deviation**: The market reacts to surprises. If Actual is higher than Forecast, the currency typically appreciates (and vice versa).

---

## 3. Database Schema Design

Table `economic_events`:
*   `id`: BigInteger, PK, autoincrement.
*   `uuid`: String(36), unique index.
*   `event_name`: String(150), label (e.g. "Fed Interest Rate Decision").
*   `country` / `currency`: String filters.
*   `importance`: String (high, medium, low).
*   `event_time`: DateTime (UTC timezone representation).
*   `actual` / `forecast` / `previous`: String fields storing values (e.g., "5.25%", "-0.1%").

---

## 4. Agent Architecture & Dependency Injection

The agent requires access to the active database connection to run tools. We leverage **Pydantic AI Context Dependencies** to inject the SQLAlchemy session safely:

```
FastAPI request context / DB session (AsyncSession)
                       │
                       ▼
       economic_calendar_agent.run(prompt, deps=session)
                       │
             ┌─────────┴─────────┐
             ▼ (Invokes tool)    ▼ (Invokes tool)
     get_upcoming_events()     get_recent_events()
             │                           │
             └─────────┬─────────────────┘
                       ▼ (Accesses ctx.deps AsyncSession)
       [EconomicCalendarRepository]
                       │
                       ▼ (Executes SQL SELECT queries)
                MySQL Database
```

## 5. API Endpoints

Mounted under `/api/v1/economic-calendar`:
*   `GET /upcoming`: Returns future events filtered optionally by currency.
*   `GET /recent`: Returns past events with actual vs forecast values.
*   `GET /high-impact`: Returns high impact releases.

## 8. Common Mistakes

*   **Hardcoding Current Time**: Using `datetime.now()` in database queries instead of `datetime.now(UTC)` without stripping tzinfo, causing offset discrepancies relative to MySQL column datatypes.
    *   *Solution*: Always use `datetime.now(UTC).replace(tzinfo=None)` or similar standard formats when working with non-tz-aware database columns.
*   **Passing session directly inside agent class constructor**: Instantiating the agent with a permanent database session.
    *   *Problem*: The session becomes stale or shared across requests, causing thread-safety failures.
    *   *Solution*: Use Pydantic AI `RunContext` to pass the session dynamically per request run call.

## 9. Interview Questions

1.  **How does Pydantic AI handle stateful context and database connections safely inside tools?**
    *   *Answer*: Rather than instantiating tools with hardcoded database session singletons, Pydantic AI implements dependency injection via `RunContext[DepsType]`. When running the agent, we pass the active, request-scoped SQL session as the `deps` argument to `agent.run(..., deps=session)`. Inside tool methods, this session is securely accessed via `ctx.deps`, ensuring clean concurrency bounds.
2.  **Why do we store past economic results (Actual vs Forecast) in the database instead of asking the LLM to remember them?**
    *   *Answer*: LLMs suffer from parametric memory gaps and hallucinations regarding precise historical numbers. In forex, a difference of 0.1% in CPI inflation dictates direction. Storing exact historical outcomes in a relational table ensures perfect numerical accuracy, which is then cited by the agent as grounded fact.
